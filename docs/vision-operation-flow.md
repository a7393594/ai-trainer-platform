# AI Trainer Platform — 願景操作流程

> 本文件從「使用者想完成什麼」出發，描述操作面板的每一步驟與預期結果。
> 不參考現有實作，純粹從產品願景推導。

---

## 第一步：建立專案

### 操作
1. 點擊「新建專案」
2. 輸入專案名稱（例如：撲克策略教練）
3. 選擇領域（或自由描述）
4. 系統啟動 Onboarding 引導

### Onboarding 引導（5-8 個問題）
系統用互動問答了解需求：
- 「這個 AI 的主要用途是什麼？」
- 「你的目標使用者是誰？」
- 「AI 應該用什麼語氣回答？」
- 「有沒有絕對不能說的內容？」
- 「需要使用工具或計算功能嗎？」

### 結果
- 系統自動產出第一版 System Prompt（可編輯）
- 自動根據領域產出 10-20 個關鍵測試問題
- 專案進入「設定中」狀態

---

## 第二步：模型選擇與比較

### 操作
1. 進入「模型比較」頁面
2. 系統列出可用模型（含價格/速度/能力標籤）
3. 勾選 3-5 個想比較的模型
4. 點擊「開始比較」

### 比較過程
- 系統用第一步產出的關鍵問題，逐一發送給每個模型
- 每個模型的回答即時呈現在並排面板中
- 顯示每次呼叫的延遲（ms）和成本（$）

### 結果
- 每個問題顯示 N 個模型的回答，並排比較
- 底部出現評估矩陣：
  - 各模型的正確率（在標記後）
  - 平均成本 per call
  - 平均延遲
  - 綜合推薦指數

---

## 第三步：人工評審與投票

### 操作
1. 逐題查看每個模型的回答
2. 每個回答可標記：✅ 正確 / ⚠️ 部分正確 / ❌ 錯誤
3. 每題可排名：哪個回答最好 → 第 1, 2, 3...
4.（可選）點擊「AI 輔助評審」讓最強模型自動打分

### 結果
- 評估矩陣即時更新
- 系統根據 正確率 × 成本 × 延遲 自動推薦一個模型
- 使用者可接受推薦或手動選擇

---

## 第四步：選定模型 + 概念差分析

### 操作
1. 點擊「選定此模型」（例如選了最便宜但夠用的 Haiku）
2. 系統自動執行概念差分析

### 概念差分析
系統自動找出：
- 「選定模型答錯，但其他模型答對」的所有題目
- 將這些題目聚類為弱點類別（例如：「錦標賽策略」「多街規劃」）

### 結果
- 顯示概念差報告：
  - 弱點題目列表（紅色標記）
  - 每題：選定模型的錯誤回答 vs 正確模型的回答
  - 弱點分類標籤
- 每個弱點旁有三個補齊按鈕：
  - 「→ 加入知識庫」（RAG 即時補齊）
  - 「→ 修正 Prompt」（產出修正建議）
  - 「→ 加入測試」（持續監控）

---

## 第五步：初步打磨

### 操作
1. 對每個概念差，選擇補齊方式（或一鍵全部補齊）
2. 系統自動執行：
   - RAG：正確回答寫入知識庫，下次 AI 能檢索到
   - Prompt：在 System Prompt 加入弱點提醒
   - Eval：弱點題自動加入測試案例

### 結果
- 知識庫新增 N 篇文件
- Prompt 新版本產出（待確認）
- Eval 測試案例新增 N 題
- 系統自動重跑一次評估 → 對比打磨前後的分數

---

## 第六步：正式對話訓練

### 操作
1. 進入「對話」頁面
2. 開始與 AI 對話（用真實使用者的問題）
3. 每次 AI 回答後，可給回饋：
   - 👍 正確
   - ⚠️ 部分正確（可提供修正文字）
   - 👎 錯誤

### AI 互動特色
- AI 回覆中如果包含問題，自動顯示互動 Widget：
  - 選擇題（A/B/C/D）
  - 排序題（拖曳排列）
  - 簡答框
- 可附加圖片或手牌紀錄給 AI 分析
- AI 可自動調用已註冊的工具進行計算

### 結果
- 每次對話都在累積訓練數據
- 回饋自動進入優化管線

---

## 第七步：自動優化迴圈

### 觸發條件
- 累積 3+ 筆負面回饋

### 自動流程
1. AI 分析所有負面回饋，找出模式
2. 產出 Prompt 優化建議（包含修改理由）
3. 使用者審核 → 接受/拒絕
4. 接受後自動產出新版本 Prompt
5. 自動跑一輪評估
6. 比對新版本 vs 舊版本的分數
7. 如果回歸（新版更差）→ 發出警告

### 結果
- Prompt 版本持續迭代（v1 → v2 → v3...）
- 每個版本有評估分數
- 分數趨勢圖（越來越高 = 越來越好）

---

## 第八步：工具串接

### 操作
1. 進入「工具管理」
2. 點擊「註冊工具」
3. 填入：
   - 工具名稱（如：勝率計算器）
   - 描述（讓 AI 知道何時該用）
   - API 端點 URL
   - 參數 Schema（JSON）
4. 儲存並測試

### 結果
- AI 在對話中會自動判斷何時需要使用工具
- 呼叫工具端點 → 取得結果 → 整合到回覆中
- 使用者感受不到中間過程，只看到帶數據的完整回覆

---

## 第九步：外部串接（給其他人用）

### Embed（嵌入式）
1. 進入「整合」頁面
2. 建立 Embed Token
3. 取得 iframe 嵌入碼
4. 貼到任何網站 → 出現聊天視窗

### API Key（程式接入）
1. 建立 API Key（sk_live_ 格式）
2. 設定 IP 白名單 + 速率限制
3. 用 REST API 呼叫：
   - POST /public/v1/chat → 取得回覆
   - GET /public/v1/sessions → 列出對話

### 結果
- 外部使用者的對話也經過同一套 Prompt + RAG + 工具
- 每個 Token/Key 隔離在自己的專案範圍內
- 使用量自動記錄（成本追蹤可見）

---

## 第十步：觀察與持續優化

### 操作
1. 進入「使用分析」查看：
   - 對話量趨勢
   - 回饋分佈（正確率是否提升）
   - 工具調用頻率
2. 進入「成本追蹤」查看：
   - 總花費
   - 各模型成本佔比
   - 每日成本趨勢
3. 進入「評估」查看：
   - 最新跑分
   - 歷史分數趨勢
   - 哪些類別最弱

### 持續迴圈
- 發現新弱點 → 加入知識庫/修正 Prompt
- 回饋累積足夠 → 匯出 JSONL → 微調模型
- 微調後替換原模型 → 成本降低但品質不變
- 重新跑評估 → 確認效果

---

## 第十一步：深度打磨（微調）

### 操作
1. 進入「微調訓練」頁面
2. 查看：原始配對數 / 清洗後配對數 / 是否達標
3. 預覽訓練數據
4. 選擇匯出格式（OpenAI / Anthropic / 通用）
5. 匯出 JSONL
6. 建立微調任務（選擇 provider + base model）

### 結果
- 取得訓練好的專用模型 ID
- 在專案設定中切換到新模型
- 成本降低 5-10 倍（自訂小模型取代大模型）
- 重跑評估確認品質

---

## 整體操作流程圖

```
新建專案 → Onboarding 引導 → 自動產出 Prompt + 關鍵問題
    ↓
模型比較 → 人工評審 → 選定模型 → 概念差分析 → 初步打磨
    ↓
對話訓練 → 回饋收集 → 自動優化建議 → 新版本 → 評估
    ↓                                         ↑
工具串接 → 外部上線 → 觀察分析 → 持續優化 ────┘
    ↓
累積數據 → 微調模型 → 替換 → 成本降低 → 持續監控
```

## 每個步驟對應的面板頁面

| 步驟 | 面板頁面 | 核心功能 |
|------|---------|---------|
| 1. 建立專案 | Chat（Onboarding 模式）| 引導問答 → 產出 Prompt |
| 2. 模型比較 | Comparison | 批次執行 + 並排比較 |
| 3. 人工評審 | Comparison（結果頁）| 標記 + 投票 + 矩陣 |
| 4. 概念差 | Comparison（Gaps tab）| 弱點分析 + 補齊按鈕 |
| 5. 打磨 | Knowledge + Prompts + Eval | 三路補齊 |
| 6. 對話訓練 | Chat | 回饋 + Widget + 工具 |
| 7. 自動優化 | Prompts | 建議 → 審核 → 新版本 |
| 8. 工具串接 | Tools | 註冊 + 測試 |
| 9. 外部串接 | Integrations | Token + API Key |
| 10. 觀察 | Analytics + Usage | 統計 + 成本 |
| 11. 微調 | Finetune | 匯出 + 訓練任務 |
